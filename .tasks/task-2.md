# Task 2 — Admin Refund Funnel for Tour Operator Statements

## 1) Бизнес контекст для задачи
Нужно сделать админский отчет по воронке возвратов Туроператору. Отчет должен помогать менеджерам быстро находить заказы, по которым у Туроператора есть сумма к возврату, но еще не подано заявление на возврат.

Основная рабочая задача пользователя админки:
- открыть список заказов с ненулевой `operator_payments.amount_awaiting_refund`;
- по умолчанию увидеть только записи без заявления на возврат (`have_written_statement = false`);
- отсортировать список по дате лога, когда менеджер проставил сумму к возврату у ТО;
- понять, кто проставил сумму к возврату и кто проставил заявление;
- перейти из строки в менеджерку заказа.

Итоговые поля таблицы:
- Заказ;
- Заявка;
- ТО;
- Организация;
- У ТО Сумма к возврату;
- Дата лога на возврат;
- Проставил(а) сумму к возврату;
- Заявление;
- Дата лога на заявление;
- Проставил(а) заявление на возврат.

## 2) Ключевые решения и обоснования
1. Реализовать отчет как read-only ActiveAdmin page, а не как новый CRUD-ресурс.
   Обоснование: данные собираются из уже существующих таблиц (`orders`, `operator_payments`, `operators`, `organizations`, `order_logs`) и не требуют отдельной persisted-сущности.

2. Вынести сбор данных в query object в `app/query`, например `Admin::OperatorRefundStatementsQuery`.
   Обоснование: отчетная выборка содержит несколько join-ов, фильтры и вычисляемые поля; controller/page ActiveAdmin должен только принимать параметры и рендерить результат.

3. Базовый набор записей брать из `operator_payments`:
   - `amount_awaiting_refund != 0`;
   - join к `orders`, `operators`, `organizations`;
   - по умолчанию фильтр `have_written_statement = false`.
   Обоснование: это доменная точка, где уже хранятся сумма к возврату, название заявки, ТО, `written_statement_at` и флаг заявления.

4. Дату лога на возврат и менеджера брать из `order_logs` по заказу, `kind = 'action_track'`, сообщению вида `У ТО сумма к возврату добавлено <> руб общая сумма <> руб.`.
   Обоснование: пользователь явно указал этот лог как источник сортировки и автора действия.

5. Менеджера заявления брать из `order_logs` по заказу, `kind = 'action_track'`, сообщению вида `Написано заявление на возврат 24.04.2026`.
   Обоснование: это явный бизнес-сигнал проставления заявления, отдельный от старого аналитического запроса.

6. В первом релизе реализовать только обязательный фильтр заявления:
   - заявление: `без заявления` / `с заявлением` / `все`;
   - по умолчанию `без заявления`.
   Обоснование: остальные фильтры из исходного аналитического SQL пока не нужны для первого релиза.

7. Для заказов с ТО `Level.Travel` не подтягивать данные из analytics/Aurora `dynamics`; показывать текущий `operators.name`.
   Обоснование: в приложении нет доступа к `dynamics`, а исходная Python-постобработка только переименовывала `Level.Travel` в `Level.Travel Мой агент` / `Level.Travel ГорТревел` для отображения.

8. Не добавлять новые гемы.
   Обоснование: для ActiveAdmin read-only отчета достаточно текущего стека, а добавление зависимостей запрещено без явного согласования.

9. Страница ActiveAdmin должна находиться в меню `Бухгалтерия` и называться `Возвраты ТО`.
   Обоснование: пользователь указал место и название страницы.

10. Сортировка по умолчанию: дата лога на возврат от новых к старым.
    Обоснование: менеджерам важнее видеть свежие задачи сверху.

11. При нескольких подходящих логах брать последний по времени.
    Обоснование: пользователь подтвердил правило; старый аналитический SQL этого надежно не гарантировал из-за `GROUP BY opa.id` без явного `MAX`/`ORDER BY`.
