# Task 1 — Operator Organization Refund Templates + Preview/Download Admin Page

## 1) Бизнес контекст для задачи
Нужно стандартизировать подготовку заявлений на возврат по конкретным `operator_organization`, чтобы менеджеры не копировали тексты вручную и могли быстро сформировать документ под заказ.

Основные цели:
- хранить шаблоны возвратных документов на уровне `operator_organization`;
- давать админам интерфейс редактирования шаблонов в знакомом rich-text редакторе (Jodit);
- дать отдельную рабочую страницу формирования документа:
  - выбрать заказ,
  - ввести сумму возврата,
  - автоматически подобрать шаблон по `order.operator_organization`,
  - отредактировать итоговый текст,
  - скачать документ в формате Word.

## 2) Ключевые решения и обоснования
1. Новая сущность `OperatorOrganizationRefundTemplate` с полями:
   - `operator_organization_id` (FK),
   - `name` (название шаблона),
   - `content` (HTML шаблон).
   Обоснование: простая модель, прозрачная связь с `operator_organization`, без лишней нормализации на первом этапе.

2. Поддержка переменных через простую подстановку плейсхолдеров (`{{order_number}}`, `{{refund_amount}}`, и т.п.).
   Обоснование: быстро запускается, легко расширять список переменных без ломки UI.

3. Отдельная admin-страница «генерации» вне ресурсной привязки к `refund_applications_mails`.
   Обоснование: пользователь явно запросил отдельную форму/страницу; это отдельный workflow от CRUD писем.

4. Автоподбор шаблона по `order.operator_organization` на клиенте:
   - заранее загрузить доступные шаблоны в селектор,
   - при выборе заказа найти шаблон по `operator_organization_id` и выставить его,
   - если не найден — пользователь выбирает вручную.
   Обоснование: минимальная сложность, быстрый UX, без лишних запросов.

5. Генерация `.doc`/`.docx` через HTML-ответ с корректными заголовками скачивания (Word-compatible).
   Обоснование: быстрый и надежный путь без добавления новых гемов (согласно ограничениям проекта).
